Security

Security designed into every request.

EliteAuth validates licenses, devices, versions, approved release hashes, nonces, and rotating session challenges before granting access. API responses are signed with Ed25519 and verified by the SDK.

Report a vulnerabilityIntegration guidance
01

Least-secret client design

Client applications use an API URL, App ID, and version. Supabase service-role credentials and application server secrets belong only in trusted environments.

02

Signed API responses

The Worker signs exact response payload bytes with Ed25519. SDKs pin the public key and reject modified, unsigned, stale, or nonce-mismatched responses.

03

Replay resistance

Short-lived request timestamps, unique nonces, hashed session tokens, and one-time rotating challenges reject captured request reuse.

04

Build integrity policy

Application owners can approve a release SHA-256 and require every activation and session check to match that exact build hash.

Defense in depth

Controls across the full path.

Security depends on the platform and on correct customer integration. Keep privileged credentials server-side, restrict allowed origins, use HTTPS, rotate secrets after exposure, and treat device identifiers as access signals rather than perfect identity.
