Quickstart

Five steps to a signed, replay-resistant session.

1

Hash your release build with SHA-256 and keep the exact value for the dashboard.

2

Create an application, enable build-integrity enforcement, and paste the approved hash.

3

Generate a license and choose its duration. Timed licenses begin on first successful device bind.

4

Configure an SDK with the API URL, App ID, version, and approved build hash.

5

Activate and revalidate. The SDK verifies Ed25519 signatures and rotates a one-time challenge.

POST/v1/license/activate

Activate a license.

The SDK sends a fresh nonce, a short-lived timestamp, and the approved build SHA-256. The API validates the license and returns a signed session token plus a one-time challenge.

Request

{
  "app_id": "YOUR_APP_ID",
  "license_key": "ELITE-7F2K-9XQM-4LPA",
  "hwid": "HASHED_STABLE_DEVICE_ID",
  "version": "1.0.0",
  "sdk_version": "cpp-1.1.0",
  "integrity_sha256": "APP_BUILD_SHA256",
  "timestamp": 1784642400,
  "nonce": "RANDOM_BASE64URL_NONCE"
}

Signed envelope

{
  "key_id": "eliteauth-ed25519-2026-01",
  "algorithm": "Ed25519",
  "payload": "BASE64URL_VERIFIED_JSON",
  "signature": "BASE64URL_SIGNATURE"
}

The SDK verifies the signature before reading the payload. A successful verified payload contains the session token, expiry, and the first one-time challenge.

200License authenticated and signed session created.
400Required input, nonce, timestamp, or hash is invalid.
401Invalid license.
403License state, HWID, version, or build-integrity policy rejected the request.
409Nonce replay detected.
426Client version does not match the allowed version.
POST/v1/session/check

Revalidate a session.

Each check consumes the previous challenge and returns a newly signed challenge. Reusing an old nonce or challenge is rejected.

{
  "app_id": "YOUR_APP_ID",
  "session_token": "TOKEN_FROM_ACTIVATION",

Anti-tampering

What the protection actually enforces.

Signed responses: the Worker signs the exact payload bytes with an Ed25519 private key. SDKs pin the public key and fail closed if verification fails.

Replay protection: every request has a short-lived timestamp and unique nonce; session checks also rotate a server-stored one-time challenge.

Build allowlisting: when enabled, the API compares the SDK-provided SHA-256 with the approved application build hash.

Security events: integrity mismatches, duplicate nonces, and challenge replays are stored in Supabase.

No client-side system can make a program impossible to patch on a device controlled by an attacker. This design detects ordinary modification and prevents forged or replayed API responses, but important authorization must still remain server-side.

SDKs

Use the client that fits your stack.

The package includes C#, C++, Python, JavaScript, TypeScript, Java, Go, and Rust clients. Every SDK verifies the same signed envelope, sends a nonce and timestamp, supplies the approved build hash, and rotates the one-time session challenge.

JavaScriptsdk/javascript/eliteauth.js
import { EliteAuthClient } from "./eliteauth.js";

const auth = new EliteAuthClient(
  "https://api.eliteauth.lol",
  "YOUR_APP_ID",
  "1.0.0",
  { integritySha256: "APP_BUILD_SHA256" }
);

const result = await auth.activate(licenseKey, hwid);
if (!result.success) throw new Error(result.error);

const session = await auth.checkSession(hwid);
Security guidance

Integrate defensively.

  • Never ship the application server secret or Supabase service-role key in client software.
  • Use HTTPS only and restrict the Worker ALLOWED_ORIGIN setting to the intended website origin where browser access is required.
  • Generate a stable device identifier without collecting unnecessary personal information; hash it before transmission when appropriate.
  • Keep the Ed25519 public key pinned in the SDK. Do not download a replacement key dynamically during authentication.
  • Hash the exact release artifact and update the dashboard hash whenever you publish a new build.
  • Treat client-side checks as best-effort. Keep high-value authorization decisions on infrastructure you control.
  • Rotate any secret immediately after suspected exposure and disable affected licenses or applications while investigating.

For vulnerability reports, follow the Responsible Disclosure Policy or email security@eliteauth.lol.

Deployment

Production checklist.

Run ANTI_TAMPER_MIGRATION.sql in Supabase SQL Editor.
Store SUPABASE_SERVICE_ROLE_KEY and ELITEAUTH_SIGNING_PRIVATE_JWK as encrypted Worker secrets.
Set the correct Supabase URL and allowed website origin in the Worker configuration.
Replace the Worker with worker/worker.js, deploy it, and verify GET /health plus GET /v1/keys.
Configure Supabase authentication site and redirect URLs for the production domain.
Deploy the root website files to Cloudflare Pages. Backend source paths are blocked by the included redirect rules.
Test registration, email verification, login, license activation, session validation, and HWID reset approval end to end.
